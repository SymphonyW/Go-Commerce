# Go电商微服务项目面试高频考点

## 1. 微服务架构设计

### Q：为什么选择微服务架构？
A：微服务架构具有以下优势：
- 单一职责：每个微服务只负责一个业务领域，如本项目中的认证服务、产品服务、订单服务等
- 独立部署：各服务可独立开发、测试和部署，缩短迭代周期
- 技术异构：不同服务可使用不同技术栈，如本项目中后端使用Go，前端使用React
- 弹性伸缩：可根据业务需求独立扩容，如购物车服务可根据流量单独扩展
- 故障隔离：单个服务故障不会影响整个系统，提高系统可靠性

### Q：微服务拆分的原则是什么？
A：遵循以下拆分原则：
- 按业务边界拆分：本项目按用户、商品、订单、购物车、商户等业务领域进行拆分
- 单一职责原则：每个服务只负责一个核心业务功能
- 松耦合高内聚：服务间通过gRPC通信，内部逻辑高度内聚
- 避免分布式事务：通过最终一致性和消息队列处理跨服务操作
- 服务粒度适中：避免服务过大或过小，本项目服务粒度控制在合理范围内

### Q：本项目的微服务架构具体包含哪些服务？
A：本项目包含以下微服务：
- API网关：处理所有客户端请求，路由到相应的微服务
- 认证服务：处理用户注册、登录和认证
- 产品服务：管理产品信息，支持CRUD操作
- 订单服务：处理订单创建和管理
- 购物车服务：管理用户购物车，基于Redis实现
- 商户服务：管理商户信息和商户产品

## 2. 服务通信

### Q：gRPC与RESTful API的区别？
A：
| 维度         | gRPC          | RESTful API   |
|--------------|---------------|---------------|
| 传输协议     | HTTP/2        | HTTP/1.1      |
| 数据格式     | Protobuf      | JSON          |
| 性能         | 更高，序列化效率高，传输体积小 | 一般，JSON序列化效率较低 |
| 接口定义     | 强类型，通过proto文件定义 | 弱类型，依赖文档 |
| 流式通信     | 支持双向流式通信 | 不支持 |
| 代码生成     | 自动生成客户端和服务端代码 | 需要手动实现 |
| 适用场景     | 服务间通信 | 客户端与服务端通信 |

### Q：本项目中如何使用gRPC？
A：本项目使用gRPC实现服务间通信：
- 在`api/`目录下定义各服务的proto文件
- 使用`protoc`工具生成Go代码
- 服务端实现gRPC接口
- 客户端通过生成的代码调用服务
- API网关作为前端与微服务之间的桥梁

### Q：如何保证微服务调用的可靠性？
A：
- 超时与重试机制：设置合理的超时时间，实现指数退避重试
- 熔断器模式：当服务调用失败率达到阈值时，快速失败避免级联故障
- 服务降级策略：当依赖服务不可用时，返回默认值或缓存数据
- 幂等性设计：确保重复调用不会产生副作用
- 负载均衡：在API网关层实现负载均衡，分发请求到多个服务实例

## 3. 数据管理

### Q：微服务中的数据库设计原则？
A：
- 每个微服务拥有独立数据库：本项目中各服务使用独立的数据库模式
- 避免跨服务数据库连接：服务间通过API或消息队列通信
- 使用事件驱动实现数据同步：通过RabbitMQ实现数据变更通知
- 最终一致性代替强一致性：接受短暂的数据不一致，保证系统可用性
- 数据模型设计：根据业务需求设计合理的数据模型，如用户模型、产品模型等

### Q：Redis在购物车中的应用？
A：本项目中Redis在购物车服务中的应用：
- 数据结构：使用Hash存储购物车商品，key为用户ID，field为商品ID，value为商品信息和数量
- 过期策略：设置购物车数据的过期时间，自动清理长时间未使用的购物车
- 性能优化：批量操作减少网络往返，使用Redis Pipeline提高性能
- 缓存策略：实现缓存击穿与雪崩解决方案，如布隆过滤器、缓存预热
- 数据一致性：与产品服务集成，确保购物车中的商品信息与系统保持一致

### Q：本项目中MySQL的使用？
A：
- 数据库设计：为每个服务设计独立的数据库表结构
- 索引优化：合理创建索引，提高查询性能
- 事务处理：在单个服务内使用数据库事务保证数据一致性
- 连接池：使用连接池管理数据库连接，提高性能
- 数据备份：定期备份数据库，确保数据安全

## 4. 分布式系统

### Q：分布式事务的解决方案？
A：
- 两阶段提交（2PC）：适用于强一致性场景，但性能较差
- 补偿事务（TCC）：Try-Confirm-Cancel模式，业务侵入性较大
- 本地消息表：通过消息队列实现最终一致性
- 最终一致性：接受短暂的数据不一致，通过消息队列异步处理
- 本项目实践：使用RabbitMQ实现最终一致性，处理跨服务的操作

### Q：如何实现服务发现？
A：
- 服务注册：微服务启动时向注册中心注册自身信息
- 服务发现：客户端从注册中心获取服务地址
- 健康检查：定期检查服务健康状态，剔除不健康的服务实例
- 负载均衡：根据服务实例状态分配请求
- 本项目实践：使用Docker Compose的服务名作为服务发现机制，API网关通过服务名调用微服务

### Q：分布式系统中的幂等性设计？
A：
- 唯一标识：为每个操作生成唯一ID，如订单号、请求ID
- 状态机：使用状态机管理业务流程，避免重复操作
- 乐观锁：使用版本号或时间戳实现乐观锁
- 防重表：使用数据库唯一约束防止重复操作
- 本项目实践：在订单创建、支付等关键操作中实现幂等性设计

## 5. 容器化部署

### Q：Docker容器化的优势？
A：
- 环境一致性：消除开发、测试、生产环境差异
- 快速部署：容器启动速度快，缩短部署时间
- 资源隔离：每个容器拥有独立的资源，避免相互影响
- 弹性伸缩：根据需求快速扩展或缩减容器数量
- 版本管理：容器镜像可版本化管理，方便回滚

### Q：Docker Compose的作用？
A：
- 定义多容器应用：在一个配置文件中定义所有服务
- 配置服务依赖：指定服务间的依赖关系，确保启动顺序
- 一键启动所有服务：使用`docker-compose up`启动整个系统
- 环境变量管理：通过环境变量配置服务参数
- 网络配置：自动创建网络，实现容器间通信
- 本项目实践：通过docker-compose.yml配置所有服务，包括MySQL、Redis、RabbitMQ和各微服务

### Q：本项目的部署流程？
A：
1. 环境准备：安装Docker和Docker Compose
2. 配置检查：确认docker-compose.yml中的环境变量配置正确
3. 启动服务：执行`docker-compose up -d`启动所有服务
4. 服务检查：使用`docker-compose ps`查看服务状态
5. 日志查看：使用`docker-compose logs`查看服务日志
6. 前端启动：进入frontend目录，执行`npm run dev`启动前端
7. 访问应用：通过<http://localhost:5173>访问前端应用
